# Backend Handoff - Quiz Submit + Consent Sync

Date: 2026-04-18
Owner: Backend team
Priority: P0 for quiz submit stability, P1 for consent sync API

## 1) Why this handoff exists
Frontend still sees 2 critical runtime issues in quiz flow:
1. User can answer 10/10 but `Submit all` may fail.
2. Legacy submit path and backend route contracts are inconsistent.

Frontend has already added client-side fallback and index fixes.
Backend changes below are needed to make the flow robust and deterministic.

## 2) P0 - API contract alignment for quiz submit

### 2.1 Add/confirm endpoint: `POST /api/v1/quiz/submit-answer`
Current frontend legacy `ApiService` calls:
- path: `/quiz/submit-answer`
- body (camelCase):
```json
{
  "sessionId": "<uuid>",
  "questionId": "<question_id>",
  "answerText": "<string>",
  "context": {}
}
```

Current backend route exists at `/quiz/submit` with snake_case payload.

Required action:
- Either add `/quiz/submit-answer` route, OR alias it to existing submit handler.
- Accept both camelCase and snake_case keys.

Minimum accepted request keys:
- `sessionId` or `session_id`
- `questionId` or `question_id`
- one of: `answerText`, `answer`, `selected_option`, `answers`

### 2.2 Keep `POST /api/v1/quiz/submit` compatible
This route already exists and should continue to support:
- `selected_option` for MULTIPLE_CHOICE
- `answer` for SHORT_ANSWER
- `answers` (object) for TRUE_FALSE_CLUSTER

### 2.3 Add/confirm endpoint: `POST /api/v1/quiz/submit-batch`
Frontend tries batch first, then fallback sequentially.

Required action:
- Implement real batch submit endpoint for reliability and lower latency.
- If not implemented yet, return explicit machine-readable error so fallback behavior is predictable.

Suggested request shape:
```json
{
  "sessionId": "<uuid>",
  "answers": [
    {"question_id": "q1", "answer": "A"},
    {"question_id": "q2", "answer": "2"},
    {"question_id": "q3", "answer": "{\"s1\": true, \"s2\": false}"}
  ]
}
```

Suggested success response:
```json
{
  "status": "success",
  "data": {
    "sessionId": "<uuid>",
    "totalSubmitted": 10,
    "submissionIds": ["...", "..."]
  }
}
```

## 3) P0 - DB write integrity for attempts
Backend currently persists attempts into `quiz_question_attempts`.

Required action:
- Ensure `question_template_id` generated by backend always matches an existing row in `quiz_question_template.id`.
- If not guaranteed, update mapping logic (do not rely on synthetic UUID if table uses real seeded UUIDs).
- Add logs for FK failure path including: `session_id`, `question_id`, `question_template_id`, exception.

Reason:
- FK mismatch can make submit appear successful in API but fail persistence, then break history/result consistency.

## 4) P1 - Consent sync API (optional but recommended)
DB schema already has:
- `profiles.consent_behavioral`
- `profiles.consent_analytics`

No schema migration is required for these fields.

Required backend action (recommended):
- Add endpoint to update consent flags in `profiles`.
- Example: `PATCH /api/v1/user/consent`

Suggested payload:
```json
{
  "consent_behavioral": true,
  "consent_analytics": false
}
```

Suggested response:
```json
{
  "status": "ok",
  "consent_behavioral": true,
  "consent_analytics": false,
  "updated_at": "2026-04-18T12:34:56Z"
}
```

## 5) Acceptance criteria (Done definition)
1. Answering 10/10 and pressing `Submit all` succeeds consistently.
2. No contract mismatch between `/quiz/submit-answer`, `/quiz/submit`, `/quiz/submit-batch`.
3. Attempt persistence to `quiz_question_attempts` has no FK failures in normal flow.
4. Quiz history/result API reflects all submitted answers.
5. (If consent sync added) frontend can round-trip consent flags with backend profile.

## 6) Quick backend test checklist
1. Submit single MULTIPLE_CHOICE via `/quiz/submit`.
2. Submit single SHORT_ANSWER via `/quiz/submit`.
3. Submit single TRUE_FALSE_CLUSTER via `/quiz/submit`.
4. Submit 10 answers via `/quiz/submit-batch`.
5. Force batch endpoint unavailable and verify sequential fallback still passes.
6. Verify `quiz_question_attempts` rows are inserted with valid FK.
